- **왜 2026에 부각:** (1) **모델 성능이 비슷해져** 어떤 모델이냐보다 **주변 시스템**이 경쟁력 (랭체인 예: 같은 모델, 하네스만 손보고 점수·순위 급상승). (2) 에이전트가 **데모→실무**로 가며 긴 작업에서 궤도 이탈·자기평가 과신·반복 실수 → **프롬프트만으로는 한계**. (3) **이름이 붙음** (미쉘 하시모토 2026-02, OpenAI 100만 줄 실험 등).
- **진화 단계:** **프롬프트** = 뭐라 말할까 → **컨텍스트** = 뭐를 첨부·보여줄까 → **하네스** = **어떤 사무실·규칙·교정 프로세스에서 일할까**. 하네스가 프롬프트·컨텍스트를 **포함하는 바깥 틀** (대체 아님).
- **네 기둥:** (1) **컨텍스트 엔지니어링** — 레포에 기계가 읽을 규칙·스택·금지사항, 필요한 것만 선별 로딩. **거대 단일 MD에 전부**는 실패 → **지도·주소·분리**. (2) **아키텍처 제약** — 말로 부탁이 아니라 **도구로 못 하게 막기**. 제약이 늘면 탐색 낭비↓. (3) **피드백 루프** — 자기평가 불가에 가깝다는 전제 → 테스트·린트·별도 리뷰 등 **가이드(전)·센서(후)**. (4) **엔트로피 관리** — 문서·코드 불일치·중복 → **정리 주기**.
- **실천 원칙:** 모델보다 **환경**부터 / 완벽 설계 말고 **실패 보고 구조에 반영** / **규칙은 적게** / **말 대신 린터·테스트로 강제** / 하네스도 **모델과 함께 진화**.
