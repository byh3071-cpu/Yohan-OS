# Evaluator — 비전·방향성 대조 체크리스트

대조 기준 문서: `docs/VISION-AND-REQUIREMENTS.md` (이 레포의 단일 비전 기록)

## 수행 절차 (에이전트가 맡을 역할)

1. **입력**: 이번 턴에서 수행한 작업 요약, 변경·산출물, 사용자 목표 한 줄.
2. **대조**: 아래 항목 각각에 대해 **충족 / 부분 / 불충족** 중 하나와 **한 줄 근거**를 적는다.
3. **판정**:
   - `pass`: 불충족 없음, 부분만 있으면 **후속 조치**를 한 줄 제안.
   - `revise`: 불충족 항목이 있음 → **무엇을 어떻게 고칠지** 구체적으로 나열.
   - `reject` (드물게): 비전의 **must_not**·안전·SoT 오염에 해당 → **중단 사유** 명시.

## 체크 항목

### A. 맥락·SoT (병목 1)

- [ ] 결과물이 **자체 SoT(`memory/`)** 원칙을 해치지 않았는가? (노션·툴이 최종 진실이 된 경우는 없는가?)
- [ ] **프로필·금지(`profile.yaml` must_not)** 를 위반하지 않았는가?
- [ ] **시크릿·키**를 SoT·코드·로그에 평문으로 넣지 않았는가?

### B. 하네스 (P → G → E)

- [ ] **영구 반영**(규칙·결정·아키텍처 변경)이 있다면, **생성만 하고 끝**이 아니라 **검토 단계**를 거쳤는가? (이번 턴에서 Evaluator가 그 역할)
- [ ] `append_decision` 또는 결정 로그가 필요한 변경인데 **남기지 않은 것**은 없는가?

### C. 비전 기능 축 (방향 유지)

- [ ] **수집·변환·MD 규격** 논의 시, **스키마·단일 파이프**를 깨지 않았는가?
- [ ] **MCP로 맥락 통일** 목표와 모순되는 설계(툴마다 다른 진실)를 추가하지 않았는가?
- [ ] **인스타·스레드 등**은 API 한계를 인지한 **현실적 경로**(스크린샷·링크)와 맞는가?

### D. 범위·품질

- [ ] 요청 범위를 넘는 **무분별 리팩터·문서 남발** 없이 **요청에 맞게** 끝냈는가?
- [ ] “정확도 100%” 같은 **불가능한 약속**을 새로 만들지 않았는가?

## 출력 형식 (에이전트가 마지막에 반드시 사용)

```text
## Evaluator
- 판정: pass | revise | reject
- 대조 요약: (A~D 중 핵심 2~4문장)
- 불충족 시: 수정 지시 (bullet)
```
