# 인사이트 요약 품질 규칙

`memory/ingest/insights/`에 두는 노트·교재 압축·OCR 정제·대시보드 SoT 초안은 **같은 요약 품질 기준**을 따른다. 목표는 **나중에 원문을 다시 안 열어도 판단·검색이 되게** 만드는 것이다.

---

## 1. 적용 범위

- 새 파일을 `ingest/insights/`에 둘 때 (수동, OCR 정제, URL·RSS 후속, 교재 인사이트).
- 기존 인사이트를 **재정제**할 때 (`draft` → `insight` 승격 전 필수).
- 대시보드 **SoT 초안 생성**·음성 메모 요약 등 AI가 본문을 쓸 때 (프롬프트·스킬이 이 문서와 동일한 원칙을 쓴다).

---

## 2. 요약의 층위 (무엇을 어디에 쓸지)

| 층위 | 역할 | 분량·형식 |
| --- | --- | --- |
| **제목** | 검색·목록에서 주제가 바로 보이게 | 명사형·질문형 한 줄. 모호한 말(“메모”, “정리”)만 쓰지 않기 |
| **목적** | 이 노트를 **언제** 다시 열지 | **한 문장** (상황 트리거 포함 권장) |
| **요약** | 원문 없이 **의미·결론**이 전달되게 | 아래 §3을 따름 |
| **본문** | 증거·단계·인용·표 | 유형별로 §5 참고 |

---

## 3. 요약(본문 상단) 품질 기준

### 반드시 지킬 것

- **문단 우선:** 가능하면 **2~4개의 짧은 문단**. 한 문단은 대략 2~4문장. 테마가 바뀔 때만 문단을 나눈다.
- **사실과 해석 분리:** 원문에 있는 사실 / 내 해석·추론을 한 문장 안에서 섞지 않는다. 추론이면 「추정」「해석」으로 표시하거나 불확실하면 **「확인 필요」**.
- **수치·연도·제품명:** 교재·기사·README 인용 시 **원문·인제스트와 숫자가 어긋나지 않게**. 애매하면 원문 경로를 적고 수치는 적지 않는 편이 낫다.
- **링크·경로:** 판단 근거가 되는 **원문 상대 경로** 또는 URL을 `## 출처·원문` 등에 둔다 (why-how·교재 인사이트 공통).

### 피할 것

- **텔레그래픽 불릿만으로 요약 채우기** — `- 단어:` 나열은 검색은 되지만 **맥락이 부족**해져 재정제 비용이 든다. 불릿은 본문·체크리스트에만 쓰고, **요약 블록은 문단이 기본**.
- **마크다운 취소선 충돌:** `DB-101~102`처럼 물결 하나가 연속되면 `~~`로 읽히는 경우가 있다. 범위 표기는 **대시·쉼표·「부터」** 등으로 바꾼다.
- **원문에 없는 정보 보강** — 빈 칸을 메우려고 추측하지 않는다.

---

## 4. 저장 전 체크리스트

- [ ] `목적` 한 문장으로 “언제 열람할지”가 보인다.
- [ ] `요약`이 문단형으로 읽히고, 첫 문장만 읽어도 **주장·결론**이 이해된다.
- [ ] 출처(경로·URL)가 있다.
- [ ] 수치·고유명사를 적었다면 원문과 한 번 더 맞춰 보았다.
- [ ] `status: insight`로 올리기 전에 위 항목을 만족한다 (`draft`는 초안 허용).

---

## 5. 유형별 보조 기준

- **교재·장 요약 (`modern-ai-ch*` 등):** 원문 `##` 순서를 한 번에 짚는 짧은 맵 + **요약 문단** + Yohan OS 적용 한 블록. 자세한 표·수치는 원문 우선.
- **GitHub why-how:** `memory/rules/github-repo-why-how.md`의 분량·필수 블록을 따른다. 요약 역할은 **한 줄로 하는 일 + 상황별 이득**이 담당한다.
- **OCR·짧은 메모:** `ocr-refine` 스킬 출력 형식 유지. Long일 때도 **「한 줄 요약」은 한 문장**, 배경은 문단으로 확장 가능.
