# 선별·맥락·파인만(2차 정제) — URL / GitHub / 유튜브 / OCR / RSS

`memory/` SoT 전제는 `notion-sync.md`, GitHub 짝 규칙은 `github-repo-why-how.md`와 같이 쓴다. 여기서는 **무엇을 오래 남길지**, **맥락 필드가 무엇인지**, **노션에서 쓰는 파인만·요약 구조를 SoT에서 어떻게 줄일지**만 정한다.

---

## 1. 맥락 필드가 정확히 뭔가

**맥락** = 원문(README, 기사, 자막, OCR 텍스트)만으로는 **나중에 복원 안 되는 정보**.


| 필드 (예시 이름) | 들어가는 것                                          | 왜 필요한가                       |
| ---------- | ----------------------------------------------- | ---------------------------- |
| **발견 경로**  | 인스타·쓰레드·HN·검색·친구 공유 등                           | “어디 환경에서 끌려왔는지”가 다시 찾는 루트가 됨 |
| **당시 질문**  | 캡처·저장 순간 머릿속 질문 한 줄                             | **선별·평가** 재현(왜 이걸 남겼는지)      |
| **출처 메타**  | URL, 채널, `ingested_at`, (텔레그램이면) `message_id` 등 | 맥락 왜곡 방지·추적                  |


인제스트 프론트매터에 이미 있는 것(`source_url`, `ingested_at` 등)은 **출처 맥락**에 해당한다. **부족한 것**은 보통 **발견 경로·당시 질문**(주관적 맥락)이다.

---

## 2. 발견 경로·당시 질문 — 선택 vs 장기보존 필수

**가늠:**

- **도구·절차·누적 레퍼런스** (노션 사용법, 클로드 코드 스킬 N종, 깃헙 오픈소스 카탈로 등) → 나중에 **검색 키워드가 사람 말과 다름**. 이때 **당시 질문**이 있으면 90일 뒤에도 찾는다. → **장기보존으로 분류할 때는 필수에 가깝게** 쓰는 것을 권장.
- **일회성 동기부여·감상** (기분 좋게 읽은 글, 짧은 명언 캡처) → **선택**. 없어도 되고, 있으면 평가에만 도움.

정리:


| 보존 등급                   | 발견 경로     | 당시 질문     |
| ----------------------- | --------- | --------- |
| **가벼움·일시** (불명확 가치)     | 선택        | 선택        |
| **장기보존 후보** (도구/학습/재사용) | **권장 필수** | **권장 필수** |


`status`나 태그로 등급을 표시하면 에이전트·본인 모두 동일 규칙을 적용할 수 있다 (예: `archive_tier: scratch | standard | long_term` — 이름은 취향).

---

## 3. 선별·평가 기준 (共感한 내용 정리)

다음 중 **하나라도 강하게 Yes**이면 **장기보존 후보**로 올려도 된다.

1. **90일 안에 한 번이라도 다시 찾을 것 같다.**
2. **지속가능한가** — 워크플로·습관·시스템에 **반복해서** 연결되는가 (한번 쓰고 끝이 아닌가).
3. **2번 이상 쓸 수 있는가** — 같은 자료를 **다른 맥락에서 재활용**(복붙, 인용, 비교, 가르치기)할 여지가 있는가.

전부 No에 가깝고 감상 위주면 **가벼운 등급**으로 두고, 파인만·요약도 **라이트 티어**만 적용한다.

---

## 4. 파인만·노션 스크린 구조 → SoT 2차 정제에 옮기기

노션 예시(캡처 기준)는 크게 네 가지 패턴이 있다.

1. **파인만 3단:** 쉬운 설명 · 실생활(또는 내 일) 예시 · 궁금한 점
2. **핵심 키워드 3:** 단어별 **의미 · 중요한 이유**
3. **인사이트 분석:** 핵심 인사이트 · 트렌드 이해(현재·신호·영향) · 실용적 적용(즉시·단기·장기)
4. **기사 요약:** 핵심 내용 요약 · 주요 포인트(배경·현재·향후·영향)

**문제:** 네 가지를 **매 건 전부** 채우면 과하고, 인제스트와 **중복**되기 쉽다.

**제안: 티어**


| 티어           | 대상 (예)                          | 채울 블록                                                                |
| ------------ | ------------------------------- | -------------------------------------------------------------------- |
| **Light**    | 짧은 RSS, 동기부요 위주, 가벼운 클립         | **기사 요약형**만: 핵심 1문단 + 포인트 2~~4개 **또는** 파인만 중 **쉬운 설명 + 궁금한 점 1~~2개** |
| **Standard** | GitHub `why-how`, 긴 유튜브, 중요 OCR | **파인만 3단 전부** **또는** **키워드 3** (의미+중요 이유) — 둘 중 **하나** (중복 피함)       |
| **Full**     | 장기보존 확정                         | **인사이트 분석** + **맥락 필드(발견 경로·당시 질문)** + (필요 시) Light 요약 1블록           |


**트렌드 이해 vs 주요 포인트:** 둘 다 “배경·지금·앞으로·영향”을 나눈다. SoT에서는 **한 세트만** 쓴다. 기사형이면 **주요 포인트** 헤딩, 트렌드 서술이 길면 **트렌드 이해** 헤딩 하나로 통일해도 된다.

## 5. 애매한 점·추가 논의 후보

- **full + feynman 둘 다?** → 비추. Full이면 **인사이트 분석 안에** “쉬운 한 줄”만 넣고, 별도 파인만 3단은 생략해도 된다.  
- **궁금한 점**은 나중에 검색 큐로 쓰기 좋다 — `memory/inbox`나 별도 `open-questions.md`로 빼도 됨 (미정).  
- **무결성·보안:** 민감본은 `memory/`가 아닌 저장소 정책 따름; **Archive 폴더**(`memory/archive/` 등)는 나중에 생기면 읽기 전용·드물게 갱신 원칙을 붙이면 된다.

## 6. 다른 규칙과 연결

- GitHub: `github-repo-why-how.md` — **포맷 분리 원칙**(`github=why-how`, `ocr/rss=feynman`)을 적용한다. GitHub는 why-how 구조를 기본으로 하고, 필요 시 파인만 3단을 선택적으로 추가한다.
- OCR·노션: `notion-ocr-pipeline.md`, `telegram-inbox.md`.

에이전트는 본 문서를 참고해 2차 정제 시 **티어를 사용자에게 확인**하거나, 프론트매터 `archive_tier`·`discovery`·`question_at_capture` 채움을 제안할 수 있다.
